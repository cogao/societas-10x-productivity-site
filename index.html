Productivity

Productivity 的定义不能只看代码量。这里先看行业标准框架演进，再用 Anthropic、AI 初创公司和 Societas 四因子模型建立共同语言。

行业标准框架演进

行业对 productivity 的理解已经从“活动量”逐步走向“价值流、质量、体验和业务结果”的组合衡量。

Anthropic: 从 8x 代码量到多层验证

Anthropic 报告里最醒目的数字是典型工程师的 lines merged / engineer / day 约为 2024 年的 8x。但他们也明确说明：代码行数只是粗信号，需要用质量、受控 benchmark、主观调查和真实案例一起校准。

Level 1 · 量

代码行数 8x

Lines merged / engineer / day 是客观、可自动统计的活动量信号，但容易高估真实 productivity。

Level 2 · 质

Session 成功率

按任务难度看 Claude Code session 是否无需纠正就完成任务，开放式任务成功率达到 76% 是质量信号。

Level 3 · 主观

员工调查 4x

研究团队主观估计使用 AI 后产出约 4x。主观感受不能单独作证，但能补充系统数据看不到的体验。

Level 4 · 受控

优化 benchmark 52x

在固定目标和正确性检查下，让模型反复优化同一任务，形成可跨模型、跨时间比较的标准化指标。

Level 5 · 案例

真实场景 ROI

例如大规模修复、两小时 debug、自动 review 捕获历史问题，验证 AI 是否在真实工作里减少人力和等待。

方法论

不迷信单一指标

Anthropic 的方法不是用一个数字证明 10x，而是用活动量、质量、感知、受控实验和真实案例交叉验证。

Societas: 四因子 productivity 模型

对 Societas 来说，更适合采用一个团队系统视角：Productivity = 商业效率 × 工程效率 × 产品质量 × 团队协作效率。这也对应我们后面的 4 个核心指标。

Productivity = 商业效率 × 工程效率 × 产品质量 × 团队协作效率

对应到 Metrics 模块：Outcome Velocity 看商业效率，Delivery Flow 看工程效率，Quality Stability 看产品质量，Team Flow Health 看团队协作效率。AI Level Achievement 不单独作为 productivity，而是解释 AI 对这四项改善的贡献。

Metrics

所有指标定义集中在这里：AI Level 作为参考框架，核心指标用于判断 productivity，AI Level 达成情况用于解释 AI 对团队提效的贡献，辅助验证指标用于定位原因。

核心指标

指标分成两类：4 个 high-level 核心指标判断 productivity 是否真的提升；AI Level 达成情况用来解释 AI 对这些提升的贡献。辅助验证指标用于定位原因和瓶颈。

High-level 核心指标 指标定义 怎么衡量 为什么设计
Outcome Velocity Outcome Validation Lead Time
计算：outcome validated date - committed date。按 product goal / customer problem 统计 P50/P75。		 每个 workstream 绑定一个 outcome metric；没有 outcome metric 的任务只进入辅助统计。 Productivity 最终要体现为更快交付有价值结果，而不是更多工程活动。
Delivery Flow Delivery Lead Time
计算：production-ready date - ready/committed date。按 feature、bug、incident、tech debt 分层统计 P50/P75/P90。		 按 work item 类型分层；同时保留 blocked time 作为辅助解释指标。 衡量端到端价值流是否变快，避免只优化编码局部。
Quality Stability Quality Guardrail Index
计算：Change Failure Rate、Bug Escape/Reopen Rate、MTTR 三项分别看趋势，不强行合成一个分数。		 Delivery Flow 改善时，三项质量护栏不能同步恶化；异常时 drill down 到测试、review、需求和发布原因。 10X 不能用质量透支换速度；质量是 productivity 的约束条件。
Team Flow Health Flow Health Score
计算：Blocked Time 趋势 + handoff friction pulse + AI trust/satisfaction pulse；定期观察，不做个人排名。		 系统数据 + pulse survey 结合；只看团队和流程。 防止 productivity 只看系统吞吐，忽略上下文切换、协作成本和团队信任。

AI Level Achievement

AI Level Achievement 记录 AI 从个人工具走向团队工作流的程度，用来解释 4 个核心指标的改善是否来自 AI 工作方式升级。

AI Level 达成情况 指标定义 怎么衡量 如何与 4 个核心指标一起看
AI Level Achievement L3 Workflow Adoption
计算：满足 role card、task contract、tool access、verification、human escalation 的 L3 工作流数量；并跟踪 Agent Task Success Rate。		 优先看团队协作、PR / release、质量评估和知识沉淀等 L3 工作流；同时看 Agent Task Success Rate、Human Intervention Rate、Machine-verifiable Task Ratio。 AI Level 不单独作为 productivity 成功标准；它需要和 Outcome Velocity、Delivery Flow、Quality Stability、Team Flow Health 联动，判断 AI 对团队提效的贡献。

AI Level 参考

AI Level 是 AI Level Achievement 的判断参考。先定义 AI 在团队中的工作方式，再看它是否带动核心指标改善。

L1 · 个人 AI 使用

Thought Partner

写代码、写文档、总结、头脑风暴。价值主要来自个人习惯，团队无法稳定复用，过程和结果也难以统一度量。

  • 参考指标：AI 使用人数、AI-assisted task 数、个人感知节省时间。
  • 当前判断：已普遍出现，但不能作为 10X 目标。
L2 · 个人任务辅助

Assistant

AI 接入部分上下文，帮助完成个人任务。工具已进入日常工作，但缺少统一 task contract、验证标准和团队级 dashboard。

  • 参考指标：AI-assisted PR ratio、AI review/test generation 使用量、个人任务 cycle time。
  • 当前判断：Copilot、通用 AI、PraestoClaw、AI review/test generation 均有基础。
L3 · AI as teammates

Team Workflow

Agent 承担团队级重复工作流。每个 Agent 有角色、权限、输入输出和验收标准，人类负责目标、检查点和高风险决策。

  • 参考指标：L3 workflow count、Agent task success rate、human intervention rate、machine-verifiable task ratio。
  • 阶段目标：让团队协作、PR / release、质量评估等高频流程逐步进入 L3。
L4 · AI as system

Autonomous System

AI 运行关键业务流程并持续优化。流程由 AI 持续监控、执行、验证和改进，只把异常和重大权衡升级给人。

  • 参考指标：autonomous workflow uptime、exception escalation rate、business KPI impact、governance audit pass rate。
  • 当前判断：短期不作为主攻目标，只保留为长期方向。

辅助验证指标

辅助指标不单独作为成功标准，用来解释 4 个核心指标和 AI Level 为什么变化，以及下一步该修哪里。

归属 / 用途 辅助验证指标 怎么衡量 用于解释什么 外部对照
Outcome Velocity High-value work ratio 高价值需求 / 全部完成需求；高价值由 PM 定义 success metric 后进入统计。 如果交付变快但高价值比例下降，说明只是吞吐增加，不是 productivity 增加。 AI 初创常用业务 KPI、客户满意度和收入/API 使用量校准。
Delivery Flow Cycle Time 从开发开始到 PR merge / 可验证完成；按任务类型看 P50/P75。 解释 Lead Time 变化是来自执行变快，还是排队/等待变化。 AI 初创常看 PR merge speed、实验完成周期。
Delivery Flow PR Review Time PR 创建到首次有效 review、最后一次提交到 merge；区分等待 reviewer 和等待作者。 判断 AI 加速编码后，human review 是否成为瓶颈。 Anthropic 明确提到 human review 会成为 AI 加速后的新瓶颈。
Delivery Flow Blocked Time work item 处于 blocked / waiting 状态的累计时间；记录阻塞原因。 解释慢是慢在开发、review、测试、需求澄清、环境还是跨团队依赖。 Flow / DevEx 常用 flow efficiency 和 wait time。
Quality Stability Bug Escape / Reopen Rate 逃逸到后续阶段或生产的问题比例；已关闭 bug 被 reopen 的比例。 解释质量护栏变化来自需求理解、测试覆盖、review 质量还是 AI 产出质量。 Claude 关注纠正/接管率和自动 review 缺陷捕获率。
Quality Stability Test Pass / Flaky Rate CI 测试通过率、flaky test 数量、人工测试阻塞次数。 判断质量问题是产品缺陷、测试环境不稳，还是验证体系不足。 AI 初创会把 eval coverage、CI check、benchmark 稳定性作为护栏。
AI Level 达成情况 L3 workflow count 满足 role card、task contract、tool access、verification、human escalation 的 Agent 工作流数量。 判断 AI 是否进入团队工作流，而不是停留在个人助手。 Anthropic human-agent teams 强调 roster、tools、shared context、verification。
AI Level 达成情况 Agent Task Success Rate Agent 输出无需重大返工并通过验收的比例；按任务类型和风险等级统计。 解释 L3 工作流是否真的可靠。 Anthropic 用 session success rate，并按任务难度分层。
AI Level 达成情况 Human Intervention Rate 每个 Agent 任务中人类纠偏、补上下文、接管、重做的次数和比例。 判断 AI 是减少执行负担，还是把工作转移成 prompt、解释和返工。 Anthropic 关注纠正/接管率下降，AI 初创用它判断 agent workflow 是否可规模化。
AI Level 达成情况 Machine-verifiable Task Ratio 有自动测试、脚本、rubric、checklist 或 verifier agent 可验证的任务比例。 判断是否具备给 Agent 放权的基础。 Anthropic 强调 doer-verifier harness、tests、rubrics、style guides。
AI Level 达成情况 AI-readable Backlog Ratio 具备 problem、scope、non-goals、相关系统、验收标准、验证命令、风险等级的 backlog 比例。 解释 Agent 失败是能力问题，还是任务定义和上下文问题。 Anthropic 强调 public context 和 written artifacts；AI 初创强调 hypothesis、metric、owner、eval。
Team Flow Health Meeting Load / Focus Time 会议时长、无会块、上下文切换次数；结合 pulse survey。 判断团队是否真的减少摩擦，而不是只是把压力换了形式。 SPACE / DevEx 常看 satisfaction、communication、efficiency。

Baseline WIP

先定义“跟什么时候比”和“数据从哪里来”，再讨论 10X。Baseline 当前处于 WIP，重点是先建立稳定口径，让后续趋势可追踪。

跟什么时候对比

建议用两层 baseline

  • 历史 baseline：试点启动前选择一段稳定历史窗口，作为当前系统状态。
  • 滚动 baseline：试点启动后按固定口径持续统计。
  • 对比窗口：先看方向，再看稳定趋势，最后判断是否可复制。
  • 分层对比：feature、bug、incident、tech debt、test work 分开看。
获取和现状

先把分散数据变成可重复口径

  • 工作模式是混合模式，必须先统一开始、完成、阻塞、返工等状态定义。
  • 系统数据覆盖 ADO/Jira、Repo、CI/CD、测试管理、Incident/Monitoring、Teams/Calendar，但比较分散。
  • AI 使用处于 L1/L2：Copilot、通用 AI、PraestoClaw、AI review/test generation 均有使用痕迹。
  • 当前痛点还在收集，需要用 baseline 把感受转成可讨论、可跟踪的事实。

统计原则

  • 先看团队和流程，不做个人绩效排名。
  • 先看趋势，不用单点数字做结论。
  • 先做分层，不把 feature、bug、incident 混成一个平均数。
  • 先有护栏，再追求速度。

10X Plan

10X Plan 不是单点工具替换，而是围绕团队协作、个人任务、人与 Agent 协作和质量保证四条线，把高频工作流变成 AI-readable、machine-verifiable、可复盘、可复制的团队操作系统。

团队协作加速

信息透明、SOP、会议和知识流

  • 团队信息透明度：PraestoClaw 进入 group chat，读取 group context、team memory、company background。
  • 主动访问与被动通知：让关键项目状态、决策、风险自动浮出，而不是靠人到处问。
  • Team SOP / RoB / PMO：梳理团队 SOP，并和 workflow 集成。
  • 会议高效：会前拷问出参会人想清楚“为什么开会”，会议只讨论和决策。
  • 知识共享：把项目结论、决策、复盘和 FAQ 沉淀为可搜索上下文。

关联核心指标：Team Flow Health、Delivery Flow、Outcome Velocity。
关联 AI Level：L2 -> L3，PraestoClaw 从个人问答进入团队工作流。

个人加速

Dev / PM 的个人任务效率

  • Dev 痛点收集：LLM 太慢、等待时间长、pipeline 时长、上下文缺失。
  • 项目 Agent friendly：为项目任务定义 skill / workflow，保证结果正确，而不是只记录过程步骤。
  • Code / Doc：代码理解、文档生成、变更说明、release note 自动化。
  • 代码 release 现状：梳理哪些环节可由 Agent 准备材料、检查清单和风险摘要。
  • PM 任务加速：需求整理、用户反馈聚类、方案草稿、状态同步和决策材料。

关联核心指标：Delivery Flow、Team Flow Health。
关联 AI Level：L1 -> L2，先提升个人任务效率，再沉淀为团队 workflow。

个人与 Agent 协作加速

从 Copilot 到项目 Agent

  • 收集个人和 Agent 协作痛点：哪些任务 AI 能做一部分，哪些因为上下文、工具、权限或验证卡住。
  • 明确哪些工作适合交给 Agent：manual tester 辅助事项、PR 阶段检查、变更风险说明、项目状态整理。
  • Skill / Workflow 设计原则：以“结果正确”为目标，而不是要求 Agent 复刻人类步骤。
  • Agent 输出必须带验证依据：测试、rubric、checklist、diff summary、risk summary。

关联核心指标：Delivery Flow、Quality Stability、Team Flow Health。
关联 AI Level：L2 -> L3，形成可重复的 Agent task contract。

产品质量保证

Self-eval、online eval、manual test

  • Self-eval + online eval：让功能、模型、Agent 输出尽量先被自动化评估。
  • Manual test：保留 release 阶段人工测试，重点验证高风险路径和用户体验。
  • PR / release 阶段质量清单：测试覆盖、回归风险、已知问题、rollback 条件。
  • 将质量信号接入 dashboard：change failure、bug escape、reopen、test pass、flaky、manual test blockers。

关联核心指标：Quality Stability、Outcome Velocity。
关联 AI Level：L3 的前提是 machine-verifiable task ratio 提升。

阶段一

诊断与任务建模

  • 收集团队协作、Dev、PM、Test、release 的具体痛点。
  • 把痛点映射到 4 个核心指标和 AI Level。
  • 识别哪些任务适合个人 AI，哪些适合团队 Agent workflow。
  • 定义每类 Agent task 的输入、输出、验证方式和升级条件。
阶段二

试点工作流

  • PraestoClaw in group chat：团队上下文、团队记忆、公司背景。
  • PR / release workflow：风险摘要、测试建议、release note、manual test 辅助。
  • PM / Dev workflow：需求澄清、状态同步、文档和决策记录。
  • Quality workflow：self-eval、online eval、manual test 信号接入。
阶段三

固化和扩展

  • 沉淀 SOP、workflow、agent role card、checklist。
  • 把成功工作流接入 baseline dashboard。
  • 复盘哪些指标改善来自 AI，哪些仍是流程或组织瓶颈。
  • 形成可推广到其他团队的样板间 playbook。

Human-Agent Team 规则

  • Public context：关键讨论、决策、项目状态进入 Teams 项目流。
  • Agent role card：每个 Agent 明确职责、权限、工具和升级条件。
  • Task contract：每类任务有输入模板、输出格式和验收标准。
  • Doer-verifier：Agent 执行后先由验证机制检查，再进入人工 review。

可见产出

  • 一套 Societas productivity 指标和 baseline dashboard。
  • 团队协作、个人任务、Agent 协作、质量保证四类工作流实践。
  • PraestoClaw 进入 Teams 项目流的实践方式。
  • 可推广到其他团队的 playbook：SOP、workflow、模板、角色、检查表和复盘方法。
