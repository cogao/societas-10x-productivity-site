行业对 productivity 的理解已经从“活动量”逐步走向“价值流、质量、体验和业务结果”的组合衡量。

Anthropic: 从 8x 代码量到多层验证

Anthropic 报告里最醒目的数字是典型工程师的 lines merged / engineer / day 约为 2024 年的 8x。但他们也明确说明：代码行数只是粗信号，需要用质量、受控 benchmark、主观调查和真实案例一起校准。

Level 1 · 量

代码行数 8x

Lines merged / engineer / day 是客观、可自动统计的活动量信号，但容易高估真实 productivity。

Level 2 · 质

Session 成功率

按任务难度看 Claude Code session 是否无需纠正就完成任务，开放式任务成功率达到 76% 是质量信号。

Level 3 · 主观

员工调查 4x

研究团队主观估计使用 AI 后产出约 4x。主观感受不能单独作证，但能补充系统数据看不到的体验。

Level 4 · 受控

优化 benchmark 52x

在固定目标和正确性检查下，让模型反复优化同一任务，形成可跨模型、跨时间比较的标准化指标。

Level 5 · 案例

真实场景 ROI

例如大规模修复、两小时 debug、自动 review 捕获历史问题，验证 AI 是否在真实工作里减少人力和等待。

方法论

不迷信单一指标

Anthropic 的方法不是用一个数字证明 10x，而是用活动量、质量、感知、受控实验和真实案例交叉验证。

Societas: 四因子 productivity 模型

对 Societas 来说，更适合采用一个团队系统视角：Productivity = 商业效率 × 工程效率 × 产品质量 × 团队协作效率。这也对应我们后面的 4 个核心指标。

Productivity = 商业效率 × 工程效率 × 产品质量 × 团队协作效率

对应到 Metrics 模块：Outcome Velocity 看商业效率，Delivery Flow 看工程效率，Quality Stability 看产品质量，Team Flow Health 看团队协作效率。AI Level Achievement 不单独作为 productivity，而是解释 AI 对这四项改善的贡献。

所有指标定义集中在这里：每个核心指标都要有清晰事件口径、数据源、统计方法和验证规则。核心指标判断 productivity 是否提升，AI Level Achievement 解释 AI 对这些提升的贡献，辅助验证指标用于定位原因。

核心指标

4 个 high-level 核心指标只看团队和流程，不做个人排名。每个指标都需要先定义 measurement unit，再定义开始事件、结束事件、过滤条件和判读规则。

High-level 核心指标 可测量定义 事件口径 / 数据源 统计与分层 验证与判读
Outcome Velocity Outcome Validation Lead Time
计算：outcome validated date - committed date。Measurement unit 是 product goal / customer problem，不是单个 PR 或任务。		 committed date = 团队接受目标、明确 owner、success metric 和范围的时间；outcome validated date = success metric 达标、客户反馈确认或 PM 验收记录完成的时间。数据来自 ADO/Jira goal、PM validation log、产品 telemetry、客户反馈。 按 product goal、customer problem、workstream 统计 P50/P75，并记录未绑定 outcome metric 的 work item 占比。取消、暂停、范围重写的目标单独标记，不混入完成样本。 只有当 Outcome Validation Lead Time 下降，同时 high-value work ratio 不下降、质量护栏不恶化，才判断业务效率提升。没有 success metric 的工作不进入核心指标，只进入 backlog 质量问题。
Delivery Flow Delivery Lead Time
计算：production-ready date - ready/committed date。Measurement unit 是具备验收标准的 work item。		 ready/committed date = work item 满足 problem、scope、验收标准、依赖和风险等级，团队承诺进入执行；production-ready date = PR 合入、必要 CI 通过、测试结论可用、release note / rollback 信息齐备。数据来自 ADO/Jira、Repo、PR、CI/CD、测试系统。 按 feature、bug、incident、tech debt、test work 分层统计 P50/P75/P90。拆出 queue time、active dev time、review time、test time、blocked time，避免只看总体平均数。 Delivery Flow 改善必须伴随 Quality Stability 稳定或改善。如果只是把任务切碎、延后测试、增加返工，不能算 productivity 提升。
Quality Stability Quality Guardrail Index
不强行合成一个分数，分别看 Change Failure Rate、Bug Escape/Reopen Rate、MTTR 三项趋势。		 Change Failure Rate = 造成 rollback、hotfix、incident 或紧急修复的变更数 / 总变更数。Bug Escape/Reopen Rate = 逃逸到后续阶段或已关闭后 reopen 的 bug 数 / 已关闭或已发布 work item。MTTR = service restored time - incident detected time。数据来自 release、incident、bug、CI/CD、test run。 按 severity、产品区域、变更类型、release 类型分层。看趋势和分布，不只看均值；P90 和高 severity 问题单独复盘。 质量护栏任一项持续恶化时，Delivery Flow 的改善不能单独作为成功。异常要 drill down 到需求清晰度、测试覆盖、review 质量、AI 产出质量和发布流程。
Team Flow Health Flow Health Score
由 blocked time ratio、handoff friction pulse、AI trust/satisfaction pulse、meeting load / focus time 共同解释，不做个人排名。		 blocked time ratio = work item 处于 blocked/waiting 的时间 / delivery lead time。Handoff friction 和 AI trust 通过 pulse survey 采集。Meeting load / focus time 来自 Teams/Calendar 聚合视图。 按团队、workstream、work item 类型看趋势；只展示聚合结果。定性反馈需要映射到阻塞原因：需求、依赖、review、环境、权限、AI 上下文、验证不足。 Team Flow Health 改善必须表现为阻塞和交接摩擦下降，同时团队对 AI 输出的信任不下降。若速度变快但满意度下降、会议负担上升，需要视为不可持续。

AI Level Achievement

AI Level Achievement 记录 AI 从个人工具走向团队工作流的程度。它是解释性指标，不单独证明 productivity 成功，必须和 4 个核心指标联动。

AI Level 达成情况 可测量定义 采集口径 验证与联动
AI Level Achievement L3 Workflow Adoption
计算：满足 role card、task contract、tool access、verification、human escalation 五项条件的 L3 工作流数量。配套看 Agent Task Success Rate、Human Intervention Rate、Machine-verifiable Task Ratio。		 每个候选 workflow 建一张 workflow card，记录触发条件、输入模板、Agent 权限、输出格式、验收方式、升级条件、日志位置。Agent 任务按 task id 记录开始、结束、结果、人工介入次数和验证证据。 AI Level 提升只有在核心指标同步改善或解释瓶颈变化时才有意义：例如 L3 PR workflow 增加后，PR Review Time 下降且 Quality Stability 不恶化；L3 会议/PMO workflow 增加后，Team Flow Health 改善。

AI Level 参考

AI Level 是 AI Level Achievement 的判断参考。先定义 AI 在团队中的工作方式，再看它是否带动核心指标改善。

L1 · 个人 AI 使用

Thought Partner

写代码、写文档、总结、头脑风暴。价值主要来自个人习惯，团队无法稳定复用，过程和结果也难以统一度量。

  • 参考指标：AI 使用人数、AI-assisted task 数、个人感知节省时间。
  • 当前判断：已普遍出现，但不能作为 10X 目标。
L2 · 个人任务辅助

Assistant

AI 接入部分上下文，帮助完成个人任务。工具已进入日常工作，但缺少统一 task contract、验证标准和团队级 dashboard。

  • 参考指标：AI-assisted PR ratio、AI review/test generation 使用量、个人任务 cycle time。
  • 当前判断：Copilot、通用 AI、PraestoClaw、AI review/test generation 均有基础。
L3 · AI as teammates

Team Workflow

Agent 承担团队级重复工作流。每个 Agent 有角色、权限、输入输出和验收标准，人类负责目标、检查点和高风险决策。

  • 参考指标：L3 workflow count、Agent task success rate、human intervention rate、machine-verifiable task ratio。
  • 阶段目标：让团队协作、PR / release、质量评估等高频流程逐步进入 L3。
L4 · AI as system

Autonomous System

AI 运行关键业务流程并持续优化。流程由 AI 持续监控、执行、验证和改进，只把异常和重大权衡升级给人。

  • 参考指标：autonomous workflow uptime、exception escalation rate、business KPI impact、governance audit pass rate。
  • 当前判断：短期不作为主攻目标，只保留为长期方向。

辅助验证指标

辅助指标不单独作为成功标准，用来解释 4 个核心指标和 AI Level 为什么变化，以及下一步该修哪里。

归属 / 用途 辅助验证指标 怎么衡量 用于解释什么 外部对照
Outcome Velocity High-value work ratio 高价值需求 / 全部完成需求；高价值由 PM 定义 success metric 后进入统计。 如果交付变快但高价值比例下降，说明只是吞吐增加，不是 productivity 增加。 AI 初创常用业务 KPI、客户满意度和收入/API 使用量校准。
Delivery Flow Cycle Time 从开发开始到 PR merge / 可验证完成；按任务类型看 P50/P75。 解释 Lead Time 变化是来自执行变快，还是排队/等待变化。 AI 初创常看 PR merge speed、实验完成周期。
Delivery Flow PR Review Time PR 创建到首次有效 review、最后一次提交到 merge；区分等待 reviewer 和等待作者。 判断 AI 加速编码后，human review 是否成为瓶颈。 Anthropic 明确提到 human review 会成为 AI 加速后的新瓶颈。
Delivery Flow Blocked Time work item 处于 blocked / waiting 状态的累计时间；记录阻塞原因。 解释慢是慢在开发、review、测试、需求澄清、环境还是跨团队依赖。 Flow / DevEx 常用 flow efficiency 和 wait time。
Quality Stability Bug Escape / Reopen Rate 逃逸到后续阶段或生产的问题比例；已关闭 bug 被 reopen 的比例。 解释质量护栏变化来自需求理解、测试覆盖、review 质量还是 AI 产出质量。 Claude 关注纠正/接管率和自动 review 缺陷捕获率。
Quality Stability Test Pass / Flaky Rate CI 测试通过率、flaky test 数量、人工测试阻塞次数。 判断质量问题是产品缺陷、测试环境不稳，还是验证体系不足。 AI 初创会把 eval coverage、CI check、benchmark 稳定性作为护栏。
AI Level 达成情况 L3 workflow count 满足 role card、task contract、tool access、verification、human escalation 的 Agent 工作流数量。 判断 AI 是否进入团队工作流，而不是停留在个人助手。 Anthropic human-agent teams 强调 roster、tools、shared context、verification。
AI Level 达成情况 Agent Task Success Rate Agent 输出无需重大返工并通过验收的比例；按任务类型和风险等级统计。 解释 L3 工作流是否真的可靠。 Anthropic 用 session success rate，并按任务难度分层。
AI Level 达成情况 Human Intervention Rate 每个 Agent 任务中人类纠偏、补上下文、接管、重做的次数和比例。 判断 AI 是减少执行负担，还是把工作转移成 prompt、解释和返工。 Anthropic 关注纠正/接管率下降，AI 初创用它判断 agent workflow 是否可规模化。
AI Level 达成情况 Machine-verifiable Task Ratio 有自动测试、脚本、rubric、checklist 或 verifier agent 可验证的任务比例。 判断是否具备给 Agent 放权的基础。 Anthropic 强调 doer-verifier harness、tests、rubrics、style guides。
AI Level 达成情况 AI-readable Backlog Ratio 具备 problem、scope、non-goals、相关系统、验收标准、验证命令、风险等级的 backlog 比例。 解释 Agent 失败是能力问题，还是任务定义和上下文问题。 Anthropic 强调 public context 和 written artifacts；AI 初创强调 hypothesis、metric、owner、eval。
Team Flow Health Meeting Load / Focus Time 会议时长、无会块、上下文切换次数；结合 pulse survey。 判断团队是否真的减少摩擦，而不是只是把压力换了形式。 SPACE / DevEx 常看 satisfaction、communication、efficiency。

Baseline WIP

先定义“跟什么时候比”“用哪些事件算”“数据质量是否可信”，再讨论 10X。Baseline 当前处于 WIP，重点是先建立稳定口径，让后续趋势可追踪、可复算、可解释。

跟什么时候对比

建议用两层 baseline

  • 历史 baseline：试点启动前选择一段稳定、可解释的历史窗口，按同一口径回算 P50/P75/P90 和质量护栏。
  • 滚动 baseline：试点启动后按固定统计窗口持续更新，用来判断趋势是否稳定，而不是用单点波动做结论。
  • 对照 baseline：同类 workstream 或未引入 L3 workflow 的任务可作为参考组，但只用于解释，不做团队排名。
  • 分层对比：feature、bug、incident、tech debt、test work 分开看，避免不同工作类型互相稀释。
获取和现状

先把分散数据变成可重复口径

  • 工作模式是混合模式，必须先统一开始、完成、阻塞、返工、验证、人工介入等状态定义。
  • 系统数据覆盖 ADO/Jira、Repo、CI/CD、测试管理、Incident/Monitoring、Teams/Calendar，但比较分散。
  • 每个核心指标需要一个 metric owner，维护字段映射、排除规则、缺失数据处理方式和 dashboard 解释。
  • 当前痛点还在收集，需要用 baseline 把感受转成可讨论、可跟踪、可复算的事实。

Baseline 建线步骤

Baseline 当前不追求一次完整，先建立可重复口径、事件日志和最小可用 dashboard。

1

定义 metric contract

为每个指标写清 measurement unit、开始事件、结束事件、必填字段、排除规则、owner 和数据源。没有 contract 的指标不进入首页。

2

建立事件日志

从 ADO/Jira、Repo、CI/CD、测试、Incident、Teams/Calendar 拉取事件，统一 task id、PR id、release id、incident id 和 outcome id。

3

回算历史 baseline

按同一规则回算历史窗口，输出 P50/P75/P90、样本量、缺失率和异常样本。缺失严重的指标先标 WIP，不强行比较。

4

上线滚动 baseline

按固定统计窗口持续更新，并保留口径版本。每次指标变化都要能追溯到样本、分层、异常和口径变化。

Baseline 设计

每个核心指标都用同一套 baseline 结构：历史状态、滚动趋势、数据质量和最小可用版本。

指标 历史 baseline 怎么建 滚动 baseline 怎么看 数据质量校验 最小可用版本
Outcome Velocity 回看已完成 product goal / customer problem，补齐 committed date、success metric、validated date 和验证证据；只纳入有明确 outcome 的样本。 看 Outcome Validation Lead Time 的 P50/P75，以及无 outcome metric work item 占比。趋势改善必须结合 high-value work ratio 和质量护栏一起看。 检查 outcome id 是否能关联 work item、验证证据是否存在、success metric 是否在执行前定义、取消/暂停样本是否被单独标记。 先从少数高价值 workstream 开始，建立 outcome log；没有 telemetry 的目标允许用 PM/customer validation record，但必须可追溯。
Delivery Flow 回算 work item 从 ready/committed 到 production-ready 的 lead time，按 feature、bug、incident、tech debt、test work 分层。 看 P50/P75/P90，并拆解 queue、dev、review、test、blocked。滚动趋势用于识别瓶颈从编码转移到 review、测试或发布。 检查 ready 状态是否被稳定使用、PR/CI/release 是否能关联 work item、blocked 状态是否有原因、异常超长样本是否被解释。 先用 ADO/Jira 状态流 + PR merge + CI pass 建最小事件链，再逐步接入 release 和测试系统。
Quality Stability 回算 Change Failure Rate、Bug Escape/Reopen Rate、MTTR 的历史趋势，按 severity 和产品区域分层。 Delivery Flow 变快时同步看三项质量护栏是否恶化。高 severity、P90 MTTR 和 reopen 问题单独看。 检查 incident、bug、release、rollback、hotfix 是否能互相关联；severity 是否稳定；reopen 和 escaped bug 定义是否一致。 先接入 release incident、bug reopen、CI/test pass 三类信号，形成质量护栏首页。
Team Flow Health 回看 blocked/waiting 状态、handoff 节点、会议负担和 pulse survey 结果，建立团队级趋势，不回溯个人表现。 看 blocked time ratio、handoff friction、AI trust/satisfaction、meeting load / focus time 的方向变化，并结合访谈解释异常。 检查 blocked reason 是否结构化、pulse survey 样本是否足够、Teams/Calendar 只使用聚合数据、定性反馈是否能映射到阻塞原因。 先建立 blocked reason taxonomy + 简短 pulse survey，再接入会议负担和 focus time 聚合视图。
AI Level Achievement 盘点现有 L1/L2/L3 workflow，记录每个 workflow 是否具备 role card、task contract、tool access、verification、human escalation。 看 L3 workflow count、Agent Task Success Rate、Human Intervention Rate、Machine-verifiable Task Ratio，并与 4 个核心指标变化对齐。 检查 Agent task 是否有 task id、验证证据、人工介入记录和失败原因；避免只记录“使用了 AI”，却无法证明任务完成质量。 先选择团队协作、PR / release、质量评估三类高频 workflow 建卡和日志，形成 AI Level drill-down。

Baseline dashboard（WIP）

  • 核心视图：首页只放 Outcome Velocity、Delivery Flow、Quality Stability、Team Flow Health，以及 AI Level Achievement。
  • 每个指标显示当前值、历史 baseline、滚动 baseline、样本量、缺失率和口径版本。
  • 辅助验证：Cycle Time、PR Review Time、blocked time、bug escape、test pass、agent success、human intervention 等作为 drill-down。
  • 分层视图：feature、bug、incident、tech debt、test work、workstream、severity 分开看。

统计原则

  • 先看团队和流程，不做个人绩效排名。
  • 先看趋势和分布，不用单点数字或平均数做结论。
  • 先做分层，不把 feature、bug、incident、tech debt、test work 混成一个平均数。
  • 先有护栏，再追求速度。
  • 口径变化必须留版本，否则不能和历史 baseline 直接比较。

Gaps

Gaps 用来承接 baseline 和 10X Plan：先把团队当前痛点转成可验证的缺口，再决定哪些 workflow、工具和数据口径需要优先补齐。

Dev Gaps

Dev Gaps 先聚焦开发链路中已经暴露的等待、上下文、验证和 Agent 协作问题。每一项都需要回到 Delivery Flow、Quality Stability、Team Flow Health 或 AI Level Achievement 来验证。

Dev Gap 痛点信号 影响的核心指标 需要补的 baseline 信号 可能的 10X 方向
项目知识库不稳定 Agent 可以从 repo / 文档临时总结，但每次、每个 session、每个 Agent 都重新总结，成本高；沉淀成知识库后又担心准确性、时效性和确定性。 Delivery Flow、Team Flow Health、AI Level Achievement 知识复用率、上下文重建时间、cross-session context miss、知识库 freshness、Agent 引用来源覆盖率。 建立 enterprise / project knowledge base，要求引用来源、更新时间、适用范围和置信度；关键结论进入 team memory。
Repo / 分支上下文容易错 代码库分支太多，Agent 不容易判断实际生效分支、目标分支和当前 release 分支，容易基于错误上下文给出结论。 Delivery Flow、Quality Stability、AI Level Achievement branch mismatch count、错误 repo / branch 引用次数、Agent context correction 次数、PR target branch exception。 为任务提供 repo / branch contract；Agent 执行前先校验当前分支、目标分支、release train 和受影响模块。
不敢交给 Agent 做生产操作 Agent 有时理解错意图，有时意图正确但操作错误；涉及 prod、不可逆操作或高风险流程时，人需要反复验证。 Quality Stability、Team Flow Health、AI Level Achievement high-risk task intervention rate、human approval count、rollback / irreversible risk tag、Agent action audit pass rate。 高风险 workflow 默认 human-in-the-loop；把 prod 操作拆成 plan、dry-run、approval、execute、verify 五段。
验证 AI 产出耗时长 等待 coding agent 生成之外，还要花大量时间阅读结论、验证成果、循环调整；生产级项目很难直接放飞。 Delivery Flow、Quality Stability、Team Flow Health verification time、Agent rework loop count、human review time、claim-to-evidence ratio、verification evidence coverage。 要求 Agent 输出附带测试、证据、风险边界和复现命令；用 verifier workflow 降低人工验证成本。
从零开始的方向判断困难 新 feature / 项目前期知识调研、架构设计和 spec review 很难判断 AI 信息是否正确，代码可能没问题但方向一开始就偏了。 Outcome Velocity、Delivery Flow、Quality Stability problem framing rework、architecture review time、spec correction count、wrong-direction rework、knowledge confidence rating。 在执行前增加 Problem / Mandate Framed 和 Outcome Contract 质量门；让 Agent 输出 assumptions、alternatives 和 open questions。
Agent 自测和产品质量不足 为求速度让 Agent 根据一句话需求做功能，后续反复沟通和测试细节；Agent 甚至不能主动发现 UI 元素溢出等基础问题。 Quality Stability、Delivery Flow、AI Level Achievement Machine-verifiable Task Ratio、self-test pass rate、UI regression catch rate、manual test blocker、bug escape / reopen rate。 为 Agent 提供清晰执行路径、UI 规范、模拟运行环境、用户操作脚本和可自动判定的验收标准。
外部系统操作能力弱 第三方平台注册、后台操作、找文档、跨软件倒腾数据仍需要人手动处理，例如 Linux 上使用 hermes / openclaw 或微信、Excel、内部 OA 等。 Delivery Flow、Team Flow Health、AI Level Achievement external tool handoff count、manual operation time、tool access gap、Agent blocked by permission / UI / doc。 梳理 tool access 和 automation boundary；优先让 Agent 处理可审计、可回滚、可验证的外部系统步骤。
Workflow 难以沉淀 工作内容变动快，难形成稳定 workflow；AI 在单点任务上有效，比如写代码、review PR、fix bug，但端到端流程还难闭环。 Delivery Flow、AI Level Achievement L3 workflow count、workflow reuse rate、single-task vs end-to-end ratio、Agent Task Success Rate。 先沉淀高频、边界清楚、验证充分的 micro-workflow，再组合成团队级 L3 workflow。
创新和 debug 假设质量不足 Agent 能 review 和头脑风暴，但很少主动提出新角度；解 bug 和创新需要大胆假设、小心求证，而当前常常假设平庸、求证不足。 Outcome Velocity、Team Flow Health hypothesis diversity、experiment run count、root-cause validation rate、novel option accepted rate。 把 brainstorm 拆成 hypothesis generation、evidence search、experiment plan、verification 四步，强制 Agent 先求证再结论。
反馈闭环容易跑偏 side project 可依赖周级数据反馈后人工分析改进；让 AI 自进化时容易朝过拟合方向演进。 Outcome Velocity、Quality Stability feedback loop cycle time、overfit signal、metric guardrail breach、human correction count。 把自动优化限制在有 guardrail 的实验框架内，保留人对目标函数、样本和停止条件的控制。

PM Gaps

10X Plan

10X Plan 不是单点工具替换，而是围绕团队协作、个人任务、人与 Agent 协作和质量保证四条线，把高频工作流变成 AI-readable、machine-verifiable、可复盘、可复制的团队操作系统。

团队协作加速

信息透明、SOP、会议和知识流

  • 团队信息透明度：PraestoClaw 进入 group chat，读取 group context、team memory、company background。
  • 主动访问与被动通知：让关键项目状态、决策、风险自动浮出，而不是靠人到处问。
  • Team SOP / RoB / PMO：梳理团队 SOP，并和 workflow 集成。
  • 会议高效：会前拷问出参会人想清楚“为什么开会”，会议只讨论和决策。
  • 知识共享：把项目结论、决策、复盘和 FAQ 沉淀为可搜索上下文。

关联核心指标：Team Flow Health、Delivery Flow、Outcome Velocity。
关联 AI Level：L2 -> L3，PraestoClaw 从个人问答进入团队工作流。

个人加速

Dev / PM 的个人任务效率

  • Dev 痛点收集：LLM 太慢、等待时间长、pipeline 时长、上下文缺失。
  • 项目 Agent friendly：为项目任务定义 skill / workflow，保证结果正确，而不是只记录过程步骤。
  • Code / Doc：代码理解、文档生成、变更说明、release note 自动化。
  • 代码 release 现状：梳理哪些环节可由 Agent 准备材料、检查清单和风险摘要。
  • PM 任务加速：需求整理、用户反馈聚类、方案草稿、状态同步和决策材料。

关联核心指标：Delivery Flow、Team Flow Health。
关联 AI Level：L1 -> L2，先提升个人任务效率，再沉淀为团队 workflow。

个人与 Agent 协作加速

从 Copilot 到项目 Agent

  • 收集个人和 Agent 协作痛点：哪些任务 AI 能做一部分，哪些因为上下文、工具、权限或验证卡住。
  • 明确哪些工作适合交给 Agent：manual tester 辅助事项、PR 阶段检查、变更风险说明、项目状态整理。
  • Skill / Workflow 设计原则：以“结果正确”为目标，而不是要求 Agent 复刻人类步骤。
  • Agent 输出必须带验证依据：测试、rubric、checklist、diff summary、risk summary。

关联核心指标：Delivery Flow、Quality Stability、Team Flow Health。
关联 AI Level：L2 -> L3，形成可重复的 Agent task contract。

产品质量保证

Self-eval、online eval、manual test

  • Self-eval + online eval：让功能、模型、Agent 输出尽量先被自动化评估。
  • Manual test：保留 release 阶段人工测试，重点验证高风险路径和用户体验。
  • PR / release 阶段质量清单：测试覆盖、回归风险、已知问题、rollback 条件。
  • 将质量信号接入 dashboard：change failure、bug escape、reopen、test pass、flaky、manual test blockers。

关联核心指标：Quality Stability、Outcome Velocity。
关联 AI Level：L3 的前提是 machine-verifiable task ratio 提升。

阶段一

诊断与任务建模

  • 收集团队协作、Dev、PM、Test、release 的具体痛点。
  • 把痛点映射到 4 个核心指标和 AI Level。
  • 识别哪些任务适合个人 AI，哪些适合团队 Agent workflow。
  • 定义每类 Agent task 的输入、输出、验证方式和升级条件。
阶段二

试点工作流

  • PraestoClaw in group chat：团队上下文、团队记忆、公司背景。
  • PR / release workflow：风险摘要、测试建议、release note、manual test 辅助。
  • PM / Dev workflow：需求澄清、状态同步、文档和决策记录。
  • Quality workflow：self-eval、online eval、manual test 信号接入。
阶段三

固化和扩展

  • 沉淀 SOP、workflow、agent role card、checklist。
  • 把成功工作流接入 baseline dashboard。
  • 复盘哪些指标改善来自 AI，哪些仍是流程或组织瓶颈。
  • 形成可推广到其他团队的样板间 playbook。

Human-Agent Team 规则

  • Public context：关键讨论、决策、项目状态进入 Teams 项目流。
  • Agent role card：每个 Agent 明确职责、权限、工具和升级条件。
  • Task contract：每类任务有输入模板、输出格式和验收标准。
  • Doer-verifier：Agent 执行后先由验证机制检查，再进入人工 review。

可见产出

  • 一套 Societas productivity 指标和 baseline dashboard。
  • 团队协作、个人任务、Agent 协作、质量保证四类工作流实践。
  • PraestoClaw 进入 Teams 项目流的实践方式。
  • 可推广到其他团队的 playbook：SOP、workflow、模板、角色、检查表和复盘方法。
